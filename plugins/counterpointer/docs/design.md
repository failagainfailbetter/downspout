# counterpointer design notes

`counterpointer` follows the `cadence` architectural pattern but generates a
single complementary melodic line rather than chordal harmony.

## Source mapping

The source is always the incoming MIDI pattern. In a typical setup this may be
`bassgen`, another MIDI generator, or live/performed MIDI routed into the plugin.
The core captures note onsets, pitch activity, velocity, and timing bins inside a
transport-synced cycle.

## Initial controls

- `key`, `scale`: fallback pitch vocabulary when the input does not define a
  strong local context.
- `cycle_bars`, `granularity`: transport-synced learning/playback grid.
- `follow`: favors source-like melodic contour.
- `counter`: favors contrary or oblique motion and rhythmic answer behavior.
- `short_random`: local deterministic jitter for note choice, rests, and timing.
- `long_random`: cycle-boundary phrase mutation, similar in intent to Cadence's
  `Vary`.
- `density`: probability of output notes per segment.
- `rhythm_follow`: blends source onset timing with answer timing.
- `syncopation`: pushes output toward off-beat answer positions.
- `consonance`: weights stable intervals against the captured source note.
- `embellish`: adds one or two extra hits inside a segment, allowing the output
  line to play more often than the incoming source pattern.
- `regularity`: macro control over generated behavior. Low values increase
  effective short/long randomness and timing looseness; high values suppress
  mutation and favor more predictable timing and pitch choices.
- `register`, `span`, `gate`, `velocity_follow`: output voice shaping.
- `pass_input`, `output_channel`: wrapper-facing MIDI routing behavior.

The wrapper exposes live `MIDI In` and `MIDI Out` status parameters in addition
to phrase readiness. These are diagnostic UI signals: `MIDI In` confirms that
the host send reaches Counterpointer, while `MIDI Out` confirms either input
pass-through or generated counter-melody events leaving the plugin.

## Algorithm

Each cycle builds segment captures from incoming MIDI. At the next cycle
boundary, the core builds a `PhraseState` by scoring candidate output notes:

- scale membership;
- register fit;
- consonance against the captured source note;
- voice-leading distance from the previous output note;
- same-direction movement from `follow`;
- opposite-direction movement from `counter`;
- bounded deterministic jitter from `short_random`.

Rhythm is generated by blending the source onset bin with a complementary answer
position. `Embellish` can add extra hits at musically spaced answer positions
within the same segment. The phrase is deterministic for the same controls,
transport, input, and variation state.

When `Regularity` and `Counter` are high and both random controls are low, the
core enters a stricter fugal answering path. The learned input becomes the
subject, the generated line answers it around the dominant, and high color on
Jazz-capable scales can invert the subject intervals. This uses the existing
controls rather than adding a separate genre selector to Counterpointer.

## Implementation staging

The current implementation includes a first DPF/VST3 wrapper and UI, but the
portable core remains the main source of behavior. Host validation should focus
on MIDI routing, saved-state restore, rewind behavior, and whether the current
monophonic phrase generation feels responsive enough when driven by `bassgen`.

The UI also has a `Randomise` button that assigns random values to all visible
sliders. It is deliberately a UI gesture rather than part of the portable engine.
